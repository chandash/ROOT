clearcase server processes:

All clearcase installed machines runs albd_server and admin_server

1.)albd_server:
    => starts other clearcase services
    => setups client server comm  (cc client sends RPC calls to the cc server's albd_server to access views/VOBS)
    =>manages clearcase scheduler
    => Responds rgy related requests
    => Responds license related requests
    => Responds load balancing queries from remote clearmake processes.

2.) admin_server
    => getlog uses this process
    => Requests from CC admin console for Retriving/changing localhost's cc properties
    => RGY backup

3.) credmap_server
    =>runs when hosts configured to support local views/vobs

4.) view_server
    => albd_server starts this process
    => reads .view from .vws dir
    => stops when execute
        ct endview -server
        cc shutdown
        UNIX kill

5.) vob_server (manipulate data in the VOB storage pools)
    => albd_server starts this process
    => only process that creates/deletes VOB data containers
    => reads vob_server.conf from .vbs dir (vob_snapshot_setup)
    => stops when
            ct rmvob
            cc stoped
            SYSTEM shutdown

6.) db_server (manages VOB db transactions for clients requests)
    => services only one client at a time but handle many VOBs
    => albd_server starts/ends this process when required/timedout
    =>  create/read/modify vob data or metadata

7.) vobrpc_server
    => albd_server starts/ends this process
    => handle requests from view_server processes
    => it forwards storage pool access requests to vob_server

8.) lockmgr
    => NOT started by albd_server but started by VOB server itself(asamvob1)
    => adminstrates transaction requests to VOB DBs (Calling pgm polls lockmgr, which either grants or prohibits  access to reqested data)

Description

The recoverview command repairs a view database and the associated private storage area of a dynamic view. (A snapshot view has no private
storage in the same sense as a dynamic view has.) Typically, you use this command after a system crash or similar mishap. You may also want
to use this command to regain access to stranded view-private and checked-out files. (See "Recovering view-private and checked-out files:
View lost+found directory".)

Automatic recovery

When necessary, recoverview is invoked by a dynamic view's associated view_server process. Enter this command yourself if messages in the view
log (view_log) suggest view database corruption (for example, INTERNAL VIEW DB ERROR).

Possible data loss

recoverview uses reformatview; that is, recovery involves a dump/load of the view database. recoverview deletes the old, invalid view database,
which reformatview has renamed to db.dumped.

Depending on the state of the view database, this process may cause certain information to be lost. After a view is recovered, consult the
view log to investigate possible data loss.

Note to UNIX users: The setuid bit is always lost on files that the view owner does not own.

SPECIFYING THE SOCKET OR SHARED MEMORY FILE NAME. Default: almd. (You must not change this default.)
On UNIX systems, the lockmgr communicates with other processes through /var/adm/rational/clearcase/almd, which is a shared-memory file on
some platforms and a socket on others

–a almd
Specifies almd as the leaf name of the socket or shared memory file created by the lockmgr. Using any other name is not supported.

Any VOB owned by a user without write permission to /var/adm/rational/clearcase/almd will be inaccessible.

–f num
SPECIFYING THE NUMBER OF DATABASE FILES.  Default on UNIX: 256
Specifies the number of database files that can be open concurrently. Each VOB database consists of seven database files.
The default startup values allow the lock manager to handle a maximum of 36 VOBs on a UNIX host.

Error: Too many open databases on host (try increasing the -f argument on lockmgr command line).

When this happens, you can either move some of the VOBs to another host, or increase the value of the –f option to 7*V where V represents the number of VOBs on the host.

–u num
SPECIFYING THE NUMBER OF USERS. Default on UNIX: 256
Specifies the number of db_server or vobrpc_server processes that can concurrently request locks. Each active view requires one vobrpc_server
process for each VOB that the view accesses. In addition, various operations that change VOB metadata cause a db_server process to access the VOB through the lock manager.
Poor user response time and messages of the form:
db_VISTA database -922: lockmgr is busy in the lockmgr_log file may indicate that the value of the –u option should be raised.

On hosts that do not support the shared-memory lock manager, the –u option cannot be set higher than 1018. On hosts that support the shared-memory lock manager, the –u option is bounded only by available virtual memory.

You can compute a very approximate worst case value for –u by using the formula:
V*(N/4 + 5)
where V is the number of VOBs on the host, and N is the number of users who access those VOBs.

–q num
SPECIFYING THE SIZE OF THE REQUEST QUEUE.  Default on UNIX: 1024.
Specifies the number of lock requests for locks to be queued. The lock manager
delays queuing lock requests in excess of this value. Poor user response time
and messages of the form

db_VISTA database -922: lockmgr is busy

As a rule, this value should be no greater that five times the value of the –u option.
